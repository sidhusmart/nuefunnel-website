NueFunnel
Website

A static marketing website for nuefunnel.com — an AI-powered API assistance product. Built on the JAMStack with Astro, styled with Tailwind, content-managed through Decap CMS, and deployed to Cloudflare Pages with a custom OAuth proxy.

Tech Stack

Framework

Astro 4.x — static site generator with island architecture. Outputs pure HTML/CSS/JS with zero client-side JS by default. output: 'static' mode.

Styling

TailwindCSS 3.x with the @tailwindcss/typography plugin for MDX prose rendering. Custom theme defined in tailwind.config.mjs.

Content

MDX files inside Astro Content Collections. Schema-validated via Zod. Collections: blog, customer-stories, products.

CMS

Decap CMS (formerly Netlify CMS) — a git-based headless CMS. Served from /admin. Uses GitHub as the content backend.

Hosting

Cloudflare Pages — edge-hosted static files + serverless Pages Functions for the OAuth proxy (/functions/oauth/).

Image Processing

Sharp — Node.js image processor. Custom script (scripts/optimize-images.mjs) generates WebP variants at 400w / 800w / 1200w.

Issue Tracking

Beads (bd) — git-native, CLI-first issue tracker stored in .beads/. AI-friendly and offline-capable.

Language

TypeScript in strict mode throughout src/. Astro frontmatter, content schemas, and component props are all typed.

System Architecture

High-Level Architecture
JAMStack pattern: The site is fully pre-rendered at build time — no server renders pages on request. Content changes in Decap CMS commit MDX to GitHub, which triggers a new Cloudflare Pages build, which re-generates all static HTML.

Decap CMS

Decap CMS (formerly Netlify CMS) is a git-based, open-source headless CMS. It's served as a static SPA from /admin. It reads/writes content directly to the GitHub repository via the GitHub API — there is no separate database. The configuration lives entirely in public/admin/config.yml.

Editorial Workflow

The CMS has publish_mode: editorial_workflow enabled. This means every piece of content moves through explicit status stages. Each draft lives on its own Git branch, giving you isolated preview deploys on Cloudflare Pages.

Content Lifecycle
Preview URLs: When Decap CMS creates a draft, it commits the MDX to a cms/[collection]/[slug] branch. Cloudflare Pages automatically deploys each branch to a unique preview URL like cms-blog-my-post.nuefunnel-website.pages.dev. This lets you preview content before it goes live.

OAuth Flow

Decap CMS needs a GitHub token to read/write repository content. GitHub's CORS policy prevents the browser from doing the token exchange directly — so a server-side proxy is required. Two Cloudflare Pages Functions handle this transparently.

GitHub OAuth — Step by Step
Why a popup? Decap CMS opens /oauth in a popup window. After the callback closes, it uses window.opener.postMessage() to send the token to the parent CMS window — a standard OAuth PKCE-like handshake pattern for browser-based apps.

The two secrets (GITHUB_CLIENT_ID, GITHUB_CLIENT_SECRET) are stored as Cloudflare Pages environment variables and accessed via env in the Functions context — they are never exposed to the browser.

Build Pipeline

The production build command is npm run build, which runs three steps in sequence:

1. optimize:images
Sharp → WebP variants
→
2. astro check
TypeScript type check
→
3. astro build
SSG → dist/
The image optimization step runs before the Astro build so that OptimizedImage.astro can read the JSON metadata sidecars from public/images/optimized/ at build time.

Environment Variables

GITHUB_CLIENT_SECRET Cloudflare Pages env only (never commit) Secret for token exchange in functions/oauth/callback.js
Never commit GITHUB_CLIENT_SECRET to git. The .env file is gitignored. Set it only in the Cloudflare Pages dashboard under Settings → Environment Variables.

Image Pipeline

The CMS saves uploads to public/images/uploads/. The custom Sharp script (scripts/optimize-images.mjs) processes them into WebP variants and saves metadata JSON sidecars to public/images/optimized/.

Generated Artifacts (per image)

  • image-name-400w.webp — 400px wide variant
  • image-name-800w.webp — 800px wide variant
  • image-name-1200w.webp — 1200px wide variant
  • image-name.webp — full-resolution WebP
  • image-name.json — metadata sidecar (dimensions, paths, sizes)

Quality: 80%. Original PNG files are kept in uploads/ (not deleted).

Commands Reference

npm Scripts

npm run dev
Start Astro dev server at http://localhost:4321 with hot reload. Does not run image optimization.
npm run build
Full production build: optimize images → TypeScript check → Astro SSG. Output in dist/.
npm run preview
Serve the dist/ folder locally to test the production build before deploying.
npm run optimize:images
Run Sharp image processor standalone — useful when you've added new uploads and want to regenerate WebP variants without a full build.
npm run astro -- check
TypeScript type-check only. No ESLint is configured — use this to catch type errors.

Issue Tracking — Beads

This project uses Beads (bd) — a git-native issue tracker that stores everything in .beads/issues.jsonl, committed alongside code. It's designed for AI-assisted workflows where issues should be visible without switching to a web UI.

Storage

.beads/issues.jsonl — JSONL format, one issue per line, merged like code conflicts.


.beads/config.yaml — project-level config.


.beads/interactions.jsonl — audit log of actions.

Workflow

  • Issues sync automatically with git commits
  • Works offline, syncs on push
  • Branch-aware — issues can track which branch they belong to
  • CLI-first: bd create / show / update / close / sync
